
 
## 更新日志维护规则

1. 目标与范围
- 维护 `CHANGELOG.md`（路径：`/Users/ipo/办公/2.开发环境代码/HyFrame4.0/HyFrame4.0_Test/CHANGELOG.md`），记录每个版本变更。
- 格式遵循 Keep a Changelog 与语义化版本（SemVer）。
- 严禁直接使用 `svn log` 或 Git；仅使用项目脚本。

2. 获取日志
- 使用命令：`npm run svn:log`（脚本与仓库地址见 `package.json` 的 `scripts` 与 `config.svnUrl`）。
- 如需仓库信息：`npm run svn:info`。

3. 版本边界判定
- 以提交消息“修改版本号为 x.y.z”为起点，向前检索到上一条“修改版本号为 a.b.c”为终点；区间内提交即该版本日志。
- 版本日期以“修改版本号为 x.y.z”的提交日期为准，格式 `YYYY-MM-DD`。

4. 编写格式
- 版本标题：`## [x.y.z] - YYYY-MM-DD`。
- 类别顺序：`### Added`、`### Fixed`、`### Changed`（其他如 `### Deprecated`、`### Removed`、`### Security` 仅在需要时添加）。
- 标识符用反引号包裹：组件/指令/属性/方法/类型，如 `hy-group`、`noOperation`、`boolean`。
- 中文描述简洁明确，不写实现细节或提交人信息。

5. 分类映射规则
- Added：含“新增/添加/支持/引入”；用于新组件、指令、属性、事件、管道、用法、功能。
- Fixed：含“修复/解决/纠正/异常/错误/报错”；用于样式、国际化、交互、逻辑缺陷。
- Changed：含“调整/优化/改进/更新/重命名/统一命名/移除冗余日志”；非破坏性行为。
- Deprecated/Removed/Security：按需使用并清晰标注。

6. 工作流程
- 运行日志脚本，获取最新提交。
- 按版本边界截取提交区间，避免越界。
- 提炼并规范化描述，合并重复、统一术语与命名，补充必要上下文（组件、属性、类型、默认值）。
- 按分类映射分配到 `Added`/`Fixed`/`Changed`。
- 写入 `CHANGELOG.md`，遵循既有风格与顺序（参考 4.7.0、4.6.0 的写法）。
- 自检：类别顺序正确、标识符加反引号、无越界与冗余、中文描述精炼。

7. 破坏性变更处理
- 如存在破坏性变更（影响外部 API），在 `### Changed` 内明确标注“破坏性变更”，说明旧名 → 新名与迁移提示。

8. 禁止事项
- 不使用 Git 或直接 `svn log` 获取日志。
- 不写提交人、提交号、文件路径等实现细节。
- 不并入非该版本边界内的提交。
- 不随意更改历史版本的语义归类与格式（除纠错）。

9. 示例模板
```
## [x.y.z] - YYYY-MM-DD

### Added
- 新增 `hy-xxx` 组件/指令/属性 `propName`，类型：`Type`，默认值：`Default`，说明：用途描述

### Fixed
- 修复 `hy-yyy` 在某场景下的样式/交互/报错问题（问题简述）

### Changed
- 调整 `hy-zzz` 方法/属性命名：`oldName` → `newName`（如为破坏性变更需明确标注）
```
