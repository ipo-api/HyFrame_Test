## storybook文档修改与创建规则

# 组件文档（Storybook）统一编写规则（v1）

本规则基于项目 README 与 `hy-input.component.stories.ts`，用于统一规范所有组件的 `*.component.stories.ts` 文档编写与维护，确保风格一致、易读易用、可复制粘贴。

## 基础信息
- 技术栈：`Angular@14`、`Ionic6`、`Node.js 14.21.3`。
- 文档工具：`Storybook`，遵循 `JSDocs` 注释。
- 初始化：根目录与 `projects/frame` 均需执行 `npm install`；`npm start` 参数在 `.storybook/simple-init.js`。

## 文件与命名
- 每个组件目录均需提供一个 `*.component.stories.ts` 文件，命名使用 kebab-case，与组件文件名一致，如：`hy-input.component.stories.ts`。
- 默认导出的 `title` 采用“组件分类/组件名（中文说明）”格式，例如：`表单组件/hy-input（输入框）`。

## 导入与基础配置
- 导入顺序：
  1. Angular 核心与通用模块
  2. RxJS 模块
  3. Angular 特定模块
  4. 核心应用导入
  5. 共享模块导入
  6. 环境与配置导入
  7. 相对路径导入
- Meta 配置（必须）：
  - `component`: 指向当前组件；
  - `decorators`: 使用 `moduleMetadata({ imports: [StoriesModule], providers: [HTTP] })`（如需 HTTP）；并使用 `componentWrapperDecorator` 包裹统一 Ionic 外壳：`<ion-app>`、`<ion-header>`、`<ion-content>` 与 `<hy-card>`；
  - `argTypes`: 基于 `unit.createArgTypes('<ComponentName>')`，为枚举与布尔类型补充 `control`（如 `iconColor` 使用 `select`，`isShowLine` 使用 `boolean`）。
- 模板函数：`const Template: Story<Component> = (args: any) => ({ props: args, template: '...' })`；通过 `Template.bind({})` 导出故事并设置 `storyName` 与 `args`（如需）。

## 推荐故事面板（示例清单）
- 默认模板（不换行）：演示 `ngModel`、`onFocus`/`onBlur`/`onChange`。
- 换行模板：演示 `isWrap`、标题左对齐、`ckRequired`、`isHideBorder`。
- 验证模板（表单模式）：`hy-form` + `name` + `[(formData)]` + `(formDataChange)` + `form.formValid` + TS 示例（`previewTemplate`）。
- 自定义错误信息：`errorMessage`。
- 必填不显示小红点：`[isShowRequiredMark]="false"`。
- 标题图标样式：`icon` + `iconColor`（枚举）。
- 提示模板：`tip`。
- 只读模式：`[readonly]="true"`。
- 自定义后置模板：`[afterTemplate]` 搭配 `ng-template`（图标、单位、按钮等）。
- 表单模式 + 后置模板：`hy-form` 内使用 `afterTemplate` 的综合演示。

## parameters.preview（必须）
- 每个故事均需包含 `parameters.preview`，用于展示可复制的代码与说明：
  - `HTML代码`：示例 HTML，`language: 'html'`，`copy: true`，`format: 'html'`；
  - `TS代码`：涉及表单或逻辑时提供，使用 `previewTemplate\`` 包裹 TS，`language: 'ts'`，`copy: true`，`format: 'ts'`；
  - `使用说明`：Markdown 说明属性、事件、注意事项，`language: 'markdown'`，`format: 'markdown'`。
- 统一说明要点：
  - 非表单模式使用 `ngModel` 控制值；
  - 表单模式（`hy-form`）必须设置 `name`，禁止使用 `ngModel`；表单数据通过 `form.formData` 管理；
  - `(formDataChange)` 返回最新 `form.formData`；`form.formValid` 表示验证状态。

## API 面板（必须）
- 统一导出 `Api` 故事并设置：
  - `Api.storyName = 'API'`；
  - `Api.parameters.docs.description.story = '对应说明'` 或针对组件的描述；
  - `Api.parameters.viewport.defaultViewport = 'desktop'`；Api文档默认为 `desktop` 尺寸；
- API 内容四段式：
  1) 属性（含：属性名 / 说明 / 类型 / 默认值）；
  2) 事件（事件名 / 说明 / 类型）；
  3) 组件方法（方法名 / 说明 / 参数）；
  4) 组件特定类型或扩展说明（如 `tipType` 或插槽/模板使用说明）。
- HTML 中出现 `{` 与 `}`：必须写为 `{{ '{' }}` 与 `{{ '}' }}`，避免 Angular 插值冲突。

## hy-form 使用规范（关键）
- 在 `hy-form` 内：
  - 每个表单项必须设置唯一的 `name`；
  - 禁止使用 `ngModel`，统一使用 `form.formData`；
  - `(formDataChange)` 用于同步数据；通过 `form.formValid` 控制按钮禁用等交互。

## argTypes 控件规范
- 基于 `unit.createArgTypes('<ComponentName>')`；
- 为枚举添加 `select` 控件并提供完整枚举值；
- 为布尔类型添加 `boolean` 控件；
- 仅为需要交互演示的属性暴露控制项，避免过度暴露。

## 代码与风格
- 字符串用单引号；2 空格缩进；避免尾随空白与未使用变量；
- 使用 `const` 与不可变数据结构；避免 `any`，使用接口定义数据模型；
- 使用可选链（`?.`）与空值合并（`??`）；
- 避免直接 DOM 操作；在模板中使用 `async` 管道处理 observables；
- 采用语义化 HTML 和必要的 ARIA 属性确保可访问性。

## 提交前检查清单
- 至少包含一个基础故事与一个表单模式故事；
- 每个故事均包含完整的 `parameters.preview`；
- 提供 `Api` 故事并覆盖属性/事件/方法/扩展说明；
- 严格遵循 `hy-form` 规范（`name`、无 `ngModel`、`formData`/`formValid`）；
- `argTypes` 控件设置正确；
- 统一使用 Ionic 外壳包装；
- 所有 `{}` 已使用 `{{ '{' }}` / `{{ '}' }}` 转义；
- 文件命名与风格符合 kebab-case、单引号、2 空格缩进等标准。

# 服务文档（Storybook）统一编写规则（v1）

面向服务（如原生能力封装）的文档与组件文档不同，主要以“使用文档”为核心，展示导入方式、服务方法、数据结构、完整示例、平台支持、注意事项与最佳实践。服务文档需参考 `hy-camera.service.stories.ts` 的结构与风格。

**文件与命名**
- 服务文档文件命名为 `*.service.stories.ts`，使用 kebab-case，与服务名一致，例如：`hy-camera.service.stories.ts`。
- `export default` 的 `title` 采用“服务/子分类/服务名称”，可根据环境补充，如：`服务/真机/hy-camera 相机服务`。

**Meta 与装饰器**
- `decorators` 必须统一使用：
  - `moduleMetadata({ imports: [StoriesModule] })`；
  - `componentWrapperDecorator` 包裹 Ionic 外壳，包含 `<ion-app>`、`<ion-header>`（可设置 `hyToolbarTitle` 为服务名）、`<ion-content>`，在内容区域用 `${story}` 插入故事内容；
- 服务文档通常没有可控输入，`argTypes` 保持为空对象 `{}`。

**故事组织**
- 推荐仅导出一个故事：`Api`。
- `Api.storyName` 统一设置为 `'使用文档'`（区别于组件的 `'API'`）。
- 统一设置 `Api.parameters`：
  - `viewport.defaultViewport = 'desktop'`；
  - `docs.description.story = '<服务名> 的完整API文档'`。

**Api 模板内容（结构必须完整）**
在 `template` 中按以下顺序组织内容，使用语义化 HTML 并保持风格一致：
- 导入服务：展示 `import { HyCameraService } from 'hy-frame-mobile';`，并用 `{{ '{' }}` 与 `{{ '}' }}` 转义花括号。
- 服务方法：使用表格列出方法名、说明、参数、返回值；例如：
  - `openCamera(): Promise<HyCameraData>`；
  - `openPhotoAlbum(mediaType?: 'picture' | 'video'): Promise<HyCameraData>`。
- 数据接口：使用表格说明服务返回的数据结构，列出接口名、属性、类型与说明；例如：
  - `HyCameraData.mediaUrl: SafeResourceUrl`（用于页面显示）；
  - `HyCameraData.uploadUrl: string`（用于文件上传）。
- 完整代码示例：
  - 提供 TypeScript 组件示例，展示如何注入服务、调用方法、处理返回值与错误；
  - 提供 HTML 模板示例，展示与页面交互（如按钮、卡片、图片预览）；
  - 代码需放在 `<pre><code>...</code></pre>` 中，并对 `{}` 使用 `{{ '{' }}` 与 `{{ '}' }}` 转义。
- 平台支持：使用表格说明 iOS、Android、HarmonyOS、Web 的支持情况与权限要求。
- 使用注意事项：以强调样式盒展示关键注意点，如真机环境限制、权限申请、文件路径差异、安全 URL 与上传路径的区分、媒体类型。
- 最佳实践：列出错误处理、权限检查、用户体验、文件管理、平台适配、安全考虑等建议。
- 常见问题：以问答形式列出高频问题与答案。

# 前端服务文档（非硬件，可在 Web 端交互）统一编写规则（v1）

此类服务（如提示、消息、前端实用服务）可在 Web 端直接演示，需提供交互示例与 `parameters.preview` 的代码分栏。参考 `projects/frame/src/frame/services/common/hy-alert/hy-alert.service.stories.ts` 的结构与风格。

**文件与命名**
- 文件命名为 `*.service.stories.ts`，与服务名一致（kebab-case），例如：`hy-alert.service.stories.ts`。
- `export default` 的 `title` 采用“服务/前端/<服务名> <中文说明>”，例如：`服务/前端/hy-alert 提示服务`。

**Meta 与装饰器**
- `decorators` 统一：
  - `moduleMetadata({ imports: [StoriesModule], providers: [...] })`，如服务依赖插件或 HTTP 可在此提供（例如 `[HTTP]`）。
  - `componentWrapperDecorator` 包裹统一 Ionic 外壳（`<ion-app>`、`<ion-header>`、`<ion-content>`），在内容区域用 `${story}` 插入故事内容。
- `argTypes` 推荐使用 `unit.createArgTypes('<ServiceClassName>','service')` 创建（如：`unit.createArgTypes('HyAlertService','service')`）；若无交互控制需求，可简化为空对象 `{}`。

**故事组织（必须有交互面板）**
- 推荐至少两个面板：
  - `交互模板`：展示核心方法的交互调用（如按钮触发、回调处理）。
  - `模式/配置变体`：按照服务的不同模式或配置项拆分面板，如 `提示模板(mode=alert)`、`提示模板(mode=tip)`。
- 每个面板均需使用 `Template` 函数组织交互：
  - 通过 `args['xxx'] = () => { ... }` 绑定事件回调，模板中以 `(onClick)="xxx()"` 触发；
  - 推荐使用 `StoriesService.<service>` 进行服务调用，保持 Storybook 环境下的注入与生命周期简化。

**parameters.preview（每个面板必须包含）**
- 统一三栏：
  - `TS代码`：使用 `previewTemplate\`` 包裹 TypeScript 示例代码，`language: 'ts'`，`copy: true`，`format: 'ts'`；此处不需要对花括号转义。
  - `HTML代码`：提供对应模板示例，`language: 'html'`，`copy: true`，`format: 'html'`；如有 `{}`，需要在故事 `template` HTML 中转义为 `{{ '{' }}` 与 `{{ '}' }}`。
  - `使用说明`：以 Markdown 形式说明默认行为、重要参数、边界条件与注意事项，`language: 'markdown'`，`format: 'markdown'`。

**交互模板骨架（可复制）**

**写作要点**
- 面板命名清晰，符合“交互模板”“提示模板(mode=xxx)”等格式；
- 交互示例必须可点击演示；`parameters.preview` 提供完整 TS/HTML/说明以便开发者复制；
- 将不同模式或配置拆分为独立面板，示例代码简洁、直观；
- 在 `template` 的 HTML 中，如出现 `{}` 必须使用 `{{ '{' }}` 与 `{{ '}' }}` 转义；`parameters.preview` 的 TS 代码无需转义。

**提交前检查清单（前端服务）**
- 至少包含一个 `交互模板` 面板与一个 `模式/配置变体` 面板；
- 每个面板都包含 `parameters.preview` 的三栏代码/说明；
- 采用统一 Ionic 外壳与正确的 `title` 分类；
- `argTypes` 使用 `unit.createArgTypes('<ServiceClass>','service')` 或按需为空；
- 交互代码通过 `args` 绑定方法并能在 Storybook 中正常触发；
- HTML 中 `{}` 已转义；整体风格与命名规范一致（kebab-case、单引号、2 空格缩进）。
